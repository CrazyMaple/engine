## 三、归属修正：log/telemetry 保留在 engine

### 3.1 冲突定位

计划 §2.2 将 `log` `telemetry` 划入 gamelib 清单（21 项）；但实际代码中：

| 被依赖包 | engine 内 import 处数 | 分布 |
|---|---|---|
| `engine/log` | 69 | `actor/ cluster/ cluster/federation/ cluster/provider/{consul,etcd,k8s}/ grain/ network/ pubsub/ remote/ router/` |
| `engine/telemetry` | 3 | `remote/` |

若强行迁到 gamelib，engine 就必须 `import "gamelib/log"` → 直接违反 §2.4 的「engine 不得 import gamelib」。

### 3.2 决策

2026-04-21 用户明确选择：**保留 log/telemetry 在 engine/（修正计划归属）**。

归属结果调整为：
- engine 白名单：**13 项**（原 11 + `log` + `telemetry`）；
- gamelib 清单：**19 项**（原 21 - `log` - `telemetry`）；
- 其余约束（§2.4 依赖方向、§2.3 tool 清单）保持不变。

### 3.3 语义合理性佐证

- `engine/telemetry/trace_context.go` 包注释原文：「让 actor/、remote/、log/、dashboard/ 都能共享」——本就是跨层共享的"元"契约，放 engine 骨架层更合理；
- `engine/log` 是零依赖的薄日志层，不含任何游戏侧聚合/脱敏之外的实现，属于 engine 自用。
- 后续如有 gamelib 侧「聚合日志 + 脱敏 + 广播 sink」等更重的能力，可以新增 `gamelib/logger/`（作为 engine/log 之上的增强层），不与本次拆分冲突。
