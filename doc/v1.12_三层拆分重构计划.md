# 问鼎天下 v.1.12 · 三层拆分重构计划

> 版本定位：**精简工程骨架，奠定 engine / gamelib / tool 三层可独立演进的目录与依赖结构**
> 制定日期：2026-04-20
> 修订日期：2026-04-21（容器定位修正：保留 `code/engine/` 为 git 仓库根，三层拆分在仓库内进行）
> 参考基线：v1.11（完成度 94.4%，详见 `问鼎天下v.1.11完成度审核.md`）
> 本版本不追加功能，只做**目录重排 + module 拆分 + import 路径改写**。

## 零、容器约定（2026-04-21 修订）

`code/engine/` 自 v1.0 起即为本项目唯一 git 仓库根，**不迁移**。v1.12 的"三层拆分"发生在该仓库根内部：

```
code/engine/                       ← git 仓库根（工作区容器）
├── go.work                        ← use ( ./engine ./gamelib ./tool )
├── engine/                        ← A 类 module（Actor 骨架）
├── gamelib/                       ← B 类 module（游戏功能层）
├── tool/                          ← C 类 module（开发期工具）
└── better/                        ← 只读参考实现
```

因此：
- 新 `engine` module 的 `go.mod` 位于 `code/engine/engine/go.mod`；
- `gamelib`、`tool` 同理位于 `code/engine/<name>/go.mod`；
- 容器根 `code/engine/` 本身**不再**是 Go module，原 `code/engine/go.mod` 在 Phase 4 被删除；
- 预留的 `demo_game/`、`server/` 若落位也放在 `code/engine/` 下。

下文所有路径均以此约定为准。

## 一、目标与背景

### 1.1 现状痛点

- 当前 `code/engine/` 单 module 下聚合 40+ 顶层目录（见 `CLAUDE.md` Architecture 小节），Actor 骨架、游戏业务、运维工具混放。
- 引擎骨架（actor/remote/cluster/grain/router/pubsub）与游戏功能（scene/ecs/skill/quest/inventory/...）耦合在同一 `module engine` 下，外部项目若只想使用 Actor 能力也必须拉入整棵树。
- `better/` 的参考代码、`example/` 演示、`dashboard/` Web 面板、`stress/` 压测等工具态产物与引擎运行期代码并列，审核、打包、CI 皆受影响。
- `demo_game/`、`server/` 已在 `code/` 下预留但未落位，说明后续还会有独立游戏服务工程与引擎共存。

### 1.2 拆分目标

1. **engine 只留 Actor 相关骨架**——对标 Proto.Actor + Leaf 原生骨架规模，保持最小闭环。
2. **gamelib 承载游戏侧通用功能**——日志、配置、定时器、网关、场景、寻路、ECS、行为树、房间、技能、背包、任务、邮件、排行榜、回放、Saga、持久化、同步、定点数、热更、中间件、遥测。
3. **tool 聚合开发期工具**——代码生成、CLI、控制台、Dashboard、基准、压测、部署、TestKit、样例。
4. **better/** 参考实现移出 engine，降级为工作区级别的只读参考。
5. **demo_game/**、**server/** 作为上层游戏服务样板，依赖 gamelib + tool，不反向污染引擎。

### 1.3 非目标

- **不新增任何业务功能**。所有 API 行为保持 v1.11 语义等价。
- **不做性能优化**，不改算法与数据结构。
- **不调整 proto 消息字段**，避免跨版本兼容问题。
- **不动 better/ 内部代码**，仅做整体位置迁移。

### 2.1 `engine` 模块白名单（严格）

| 目录 | 保留原因 |
|---|---|
| `actor/` | Actor Context / PID / Mailbox / Dispatcher / Supervisor / Behavior / Pool / Priority / Stash — 引擎核心 |
| `remote/` | Endpoint / RemoteProcess / 连接池 / 健康检查 — 位置透明所需 |
| `cluster/` | Gossip / 一致性哈希 / Split-Brain / Singleton / Federation / Migration / Provider — 分布式基础 |
| `grain/` | (Kind, Identity) 虚拟 Actor — Actor 基座必备 |
| `router/` | Broadcast / RoundRobin / ConsistentHash — Actor 消息路由 |
| `pubsub/` | Topic 发布订阅 — 跨 Actor 事件分发 |
| `proto/` | actor/remote/cluster/system 四份 .proto 与注册表 — 跨节点消息契约 |
| `codec/` | JSON / Binary / Stream — `remote` 的直接依赖，不可拆出 |
| `network/` | TCP / WS / KCP 与 MsgParser — `remote` 的直接依赖（gamelib.gate 通过 import engine/network 复用） |
| `errors/` | 引擎级错误分类（ConnectError / TimeoutError / ClusterError / CodecError） |
| `internal/` | MPSC 无锁队列 — mailbox 底层 |

**排除标准**：凡带"玩法/业务/配置/日志聚合/监控仪表/压测/CLI"语义的，一律移出。

> **2026-04-21 勘误**：`log/` 与 `telemetry/` 在实施阶段被确认为 engine 骨架核心依赖（engine 内 72 处 import；若迁到 gamelib 会违反 §2.4「engine 不得 import gamelib」铁律），已修正归属到 engine 白名单。修正后 engine 白名单共 **13 项**（原 11 项 + `log` + `telemetry`）。详见 `v1.12_完成度审核.md` §三。

### 2.2 `gamelib` 归属清单

`config, timer, gate, scene, ecs, bt, syncx, fixedpoint, room, skill, inventory, quest, mail, leaderboard, replay, saga, persistence, hotreload, middleware`（**19 个**，勘误后）。

> **2026-04-21 勘误**：原清单含 `log` 与 `telemetry`（共 21 项）；实施阶段修正归属到 engine（见 §2.1 勘误段），本清单相应调整为 19 项。

- `middleware` 内的"Actor Interceptor 注入点接口"保留在 `engine/actor/`（若已在 actor 包内则无需拆）；**具体实现**（OTel/RBAC/metrics/ratelimit）全部归 gamelib。
- `gate` 依赖 `engine/network`，方向：`gamelib → engine`，不反向。
- `syncx` 依赖 `gamelib/fixedpoint`，同层内部依赖允许。

### 2.3 `tool` 归属清单

`codegen, cmd, console, dashboard, testkit, bench, stress, deploy, example`（9 个）。

- `cmd/engine` 为 CLI，之前 engine-cli 的 `doctor/gen/run/bench/...` 子命令全部进入 `tool/cmd/engine/`。
- `example/` 含 `cluster_example.go / remote_example.go / pubsub_example.go / combat_example.go / unity_demo/` 等——按类型归到 `tool/example/{actor,remote,cluster,pubsub,combat,unity}/`（仅重组，无逻辑改动）。
- `testkit` 为测试辅助 DSL，留工具层；生产代码不得 `import tool/testkit`，由 build tag `//go:build testkit` 或只在 `_test.go` 中使用保障。

### 2.4 依赖方向铁律

```
tool   ──▶ gamelib ──▶ engine
 │                         ▲
 └─────────────────────────┘  (tool 可直接依赖 engine 做 Actor 调度/Dashboard 接入)

engine   不得 import gamelib / tool
gamelib  不得 import tool
better/  不被任何 module import
demo_game/ server/  只能 import engine + gamelib + tool
```

CI 守护：在 `tool/cmd/engine/` 下新增 `doctor deps` 子命令，用 `go list -deps` 检测反向引用，任何违规 PR 直接失败。

脚本能力：
1. 预检：当前必须在 `code/engine/`（仓库根）且 `git status` 干净。
2. dry-run：仅打印将要执行的命令，不改文件。
3. `--apply` 开关：真正执行 `git mv` 与 import 改写。
4. 每个阶段独立开关：`--phase=1|2|3|4`。
5. import 改写使用 `goimports -w` + `gofmt -w` 保守方案；批量替换用 `grep -rl` 定位、`sed -i` 精准替换顶层 module 前缀。
6. 所有 `git mv` 的源/目标路径均以 `code/engine/` 为根；engine 白名单目录在 Phase 4 被 `git mv <x>/ engine/<x>/`。

具体脚本说明见同目录 `v1.12_迁移脚本说明.md` 与可执行 `split_v1.12.sh`。

## 七、风险与回滚

| 风险 | 触发场景 | 处置 |
|---|---|---|
| import 循环 | gamelib/x 误 import tool | CI `doctor deps` 拦截；本地 `go vet` 提前发现 |
| remote 编码回归 | codec 留 engine，gamelib 自定义 codec 造成序列化不一致 | 保持单一 JSON codec 作默认；proto/binary codec 作为可选参数注入 |
| mgo 依赖跨 module | MongoStorage 在 gamelib，但 example 在 tool 要用 | tool 也 require mgo 或改用 `gamelib/persistence.Storage` 接口 |
| better/ 引用泄漏 | 有 .go 文件 import `better/...` | 扫描确认当前无任何引用（CLAUDE.md 已声明），脚本自动 grep 守护 |
| TestSortByType 回归 | 已知失败项 | 列入本版本首个必修项，Phase 2 开始前修 |
| 下游 demo_game/server | 预留目录当前为空 | 保持空；Phase 6 再接入 |
| 分支合并冲突 | 拆分期间其他 feature 合入 | 冻结 main；所有变更仅走本次 PR |

**回滚预案**：`git reset --hard pre-v1.12-split` 即可完整回滚。三个 module 的 go.mod 是增量文件，删除 `code/engine/{engine,gamelib,tool,go.work}`、恢复容器根原 `code/engine/go.mod` 即可回到单 module 状态。

## 八、验收标准

> 以下命令均在容器根 `code/engine/` 下执行（git 仓库根）。

1. **构建**：`(cd engine && go build ./...) && (cd gamelib && go build ./...) && (cd tool && go build ./...)` 三条命令均零错误。
2. **测试**：三 module 测试总数 ≥ v1.11 基线（47 包，≥ 基线通过数），新失败为 0（`inventory.TestSortByType` 列入本版本修复）。
3. **依赖方向**：`tool/cmd/engine doctor deps` 通过，无 engine→gamelib / gamelib→tool 反向边。
4. **engine 白名单**：`code/engine/engine/` 顶层目录集合严格等于 §2.1 11 项。
5. **容器根整洁**：`code/engine/` 直接子项只剩 `go.work` + `go.work.sum` + `README.md` + `CLAUDE.md` + `.gitignore` + `doc/` + `better/` + 三个 module 目录；不再有 `go.mod` / `go.sum`。
6. **文档**：三份子 module `CLAUDE.md` + 一份容器 `CLAUDE.md`，合计描述长度 ≤ 原 `code/engine/CLAUDE.md`，无冗余。
7. **代码行数**：`engine/engine/` LOC 应在 **25k ± 2k** 行（v1.11 全量 ≈ 87k，engine 部分占比估 30%）。
8. **兼容性**：任何外部项目若原先 `import "engine/xxx"`（单 module 时代路径），脚本提供一份 `sed` 映射表，一键完成下游改造。
