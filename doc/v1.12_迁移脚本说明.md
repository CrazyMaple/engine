# v1.12 迁移脚本说明

配套 `split_v1.12.sh`（首次执行时位于容器根 `code/engine/`，Phase 4 完成后迁入 `code/engine/tool/migrate/`），按 `v1.12_三层拆分重构计划.md` §五 阶段落地。

> 2026-04-21 修订：容器根确认为 git 仓库根 `code/engine/`，所有路径以该根为基准；不再涉及在 `code/` 层新建 `../gamelib` 等跨仓库移动。

## 用法

```bash
cd /data/works/mindserver/qmcreator/code/engine        # 容器根 = git 仓库根

# 预览（dry-run，默认不改任何文件）
bash split_v1.12.sh                        # 全部阶段 dry-run
bash split_v1.12.sh --phase 2              # 只看 Phase 2

# 真正执行（必须干净工作区 + 显式 --apply）
bash split_v1.12.sh --phase 1 --apply
bash split_v1.12.sh --phase 2 --apply
bash split_v1.12.sh --phase 3 --apply
bash split_v1.12.sh --phase 4 --apply
```

## 阶段对照

所有路径相对容器根 `code/engine/`：

| Phase | 动作 | 可回滚 |
|---|---|---|
| 1 | 创建 `./engine/`（新子目录）、`./gamelib/`、`./tool/`、`./go.work`；生成三个 go.mod 骨架（`engine/engine/go.mod`、`gamelib/go.mod`、`tool/go.mod`）；`better/` 留在 `./better/` 原位 | `rm -rf` 新建目录即可 |
| 2 | `git mv ./<b类模块>/ ./gamelib/<b类模块>/`（21 项），批量改写 import `"engine/<b>" → "gamelib/<b>"` | `git reset --hard` |
| 3 | `git mv ./<c类模块>/ ./tool/<c类模块>/`（9 项），批量改写 import `"engine/<c>" → "tool/<c>"`；tool 内部引用 gamelib 的包同步改写 | `git reset --hard` |
| 4 | `git mv ./<a类白名单>/ ./engine/<a类白名单>/`（11 项）；删除容器根残留的 `go.mod` `go.sum`；`cd engine && go mod tidy`，同样在 gamelib、tool 下 tidy；跑 build/test；把 `split_v1.12.sh` 迁入 `tool/migrate/` | `git reset --hard` |

## Import 改写策略

- 扫描目标：所有 `*.go` 文件
- 改写规则（按优先级从上到下）：
  1. B 类 21 个目录逐一映射：`"engine/log"` → `"gamelib/log"`、…
  2. C 类 9 个目录逐一映射：`"engine/codegen"` → `"tool/codegen"`、…
  3. 子路径同步：`"engine/codegen/..."`、`"engine/config/..."` 等
- 实现：`grep -rln "engine/<src>"` 定位，`sed -i 's#engine/<src>#<newpath>#g'` 精准替换（以 `"` 或 `/` 为边界，避免误伤）
- A 类白名单 11 个目录的 import 路径**保持 `engine/xxx` 不变**（仅物理位置从 `code/engine/xxx/` 移到 `code/engine/engine/xxx/`，module 名 `engine` 不变）
- 执行顺序：先 gamelib → 再 tool → 最后 engine 自身 tidy

## 前置检查

脚本启动即校验：
1. `git status --porcelain` 为空（工作区干净）
2. `pwd` 以 `/code/engine` 结尾（即容器根）
3. `git rev-parse --abbrev-ref HEAD` 不是受保护分支（默认拒绝在 `main` 上直接执行 `--apply`，需加 `--force-main`）
4. `go.work` 与 `engine/engine/go.mod` 的存在状态与当前 phase 相符（例如 Phase 2 要求 Phase 1 产物齐备）

## 守护项

- Phase 2/3 完成后自动调用各 module 的 `go vet ./...`（工作目录切到对应 module 根）
- 发现反向依赖（engine→gamelib / gamelib→tool）时脚本立即退出并打印违规行
- `better/` 目录始终留在 `code/engine/better/`，脚本抓取所有 `import "engine/better/..."` 确认为 0 后才继续
- Phase 4 末尾校验：`code/engine/` 直接子项不含 `go.mod` / `go.sum`；`engine/engine/` 下顶层目录数 == 11

## 失败恢复

任何阶段失败：
```bash
git reset --hard pre-v1.12-split    # 回到起点
```
然后修脚本再来。不要手动部分回退，避免 import 半改写态。
